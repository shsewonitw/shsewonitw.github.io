## AI Agent란

AI Agent는 단순히 질문에 답하는 것을 넘어서 **목표를 달성하기 위해 스스로 행동을 계획하고 실행하는 시스템**이다. 사용자의 고수준 목표를 받아 이를 여러 하위 태스크로 분해하고, 필요한 도구를 호출하며, 그 결과를 바탕으로 다음 행동을 결정한다.

LLM(Large Language Model)을 에이전트의 "두뇌"로 활용하는 방식이 일반적이며, 모델이 어떤 도구를 언제 쓸지 판단한다.

---

## ReAct 패턴

에이전트의 가장 기본적인 동작 패턴은 **ReAct(Reasoning + Acting)**다.

```
Thought: 현재 상황을 분석하고 무엇을 해야 할지 추론
Action: 도구 호출 (어떤 도구를, 어떤 인자로)
Observation: 도구 호출 결과 확인
Thought: 결과를 바탕으로 다시 추론
Action: 다음 행동 결정
...
Final Answer: 최종 답변 도출
```

이 루프를 반복하면서 에이전트는 복잡한 태스크를 단계적으로 해결한다.

---

## Tool Use (Function Calling)

Tool Use는 LLM이 외부 함수나 API를 호출할 수 있도록 하는 기능이다. 모델은 텍스트를 생성하는 대신, 특정 형식의 **구조화된 출력**을 뱉어서 어떤 도구를 어떤 인자로 호출할지 표현한다.

### 동작 흐름

```
1. 사용자 메시지 + 사용 가능한 도구 목록 → LLM
2. LLM → tool_call 응답 (도구 이름 + 인자)
3. 런타임이 실제 함수 실행
4. 실행 결과 → LLM에 다시 전달
5. LLM → 최종 응답 생성
```

### 예시 (OpenAI / Anthropic 스타일)

도구 정의:
```json
{
  "name": "get_weather",
  "description": "특정 도시의 현재 날씨를 반환한다",
  "parameters": {
    "type": "object",
    "properties": {
      "city": {
        "type": "string",
        "description": "날씨를 조회할 도시 이름"
      }
    },
    "required": ["city"]
  }
}
```

LLM의 응답:
```json
{
  "type": "tool_use",
  "name": "get_weather",
  "input": { "city": "Seoul" }
}
```

실제로 `get_weather("Seoul")`을 실행하고, 결과를 다시 모델에 넘기면 모델이 자연어로 최종 답변을 만든다.

## 주요 도구 유형

| 유형 | 예시 |
|------|------|
| 검색 | 웹 검색, 벡터 DB 검색, 문서 검색 |
| 코드 실행 | Python 인터프리터, SQL 실행 |
| 외부 API | 날씨 API, 결제 API, 캘린더 API |
| 파일 I/O | 파일 읽기/쓰기, 스크린샷 캡처 |
| 다른 에이전트 | 서브 에이전트 호출 (멀티 에이전트) |

---

## 싱글 에이전트 vs 멀티 에이전트

**싱글 에이전트**는 하나의 LLM이 모든 도구를 직접 관리하며 태스크를 처리한다. 구조가 단순하지만, 태스크가 복잡해질수록 컨텍스트가 길어지고 성능이 저하될 수 있다.

**멀티 에이전트**는 여러 에이전트가 역할을 나눠 협업하는 구조다. 오케스트레이터 에이전트가 전체 플로우를 조율하고, 서브 에이전트들이 특화된 태스크(검색, 코드 작성, 검증 등)를 담당한다.

```
Orchestrator
  ├─ Research Agent  (웹 검색, 문서 요약)
  ├─ Coder Agent     (코드 생성, 실행)
  └─ Critic Agent    (결과 검증, 피드백)
```

복잡한 태스크는 멀티 에이전트로 병렬 처리하면 성능과 확장성을 높일 수 있다.

---

## 에이전트의 메모리 구조

에이전트는 네 가지 메모리 유형을 활용한다.

| 메모리 유형 | 설명 | 예시 |
|------------|------|------|
| In-context | 현재 대화창 내 정보 | 이번 세션 히스토리 |
| External | 외부 저장소에서 검색 | 벡터 DB, 관계형 DB |
| In-weights | 파인튜닝으로 내재화 | 도메인 특화 지식 |
| In-cache | KV 캐시 | 이전 프롬프트 재사용 |

실시간으로 정보를 누적해야 하는 장기 세션에서는 External 메모리를 적극 활용한다.

## 에이전트 구현 시 고려사항

**할루시네이션 방지**: 도구 결과에 근거해서 답변하도록 강제하고, 근거 없는 추론은 하지 않도록 시스템 프롬프트로 유도한다.

**무한 루프 방지**: 최대 스텝 수(max_iterations)를 설정해야 한다. 목표에 도달하지 못한 채 계속 도구를 호출하는 상황을 막기 위해서다.

**도구 오류 처리**: 도구 호출이 실패했을 때 에이전트가 재시도 또는 대안을 선택할 수 있도록 에러 메시지를 명확히 전달해야 한다.

**비용 및 레이턴시**: 도구 호출마다 LLM API 요청이 발생한다. 병렬 호출 가능한 도구는 동시에 실행해서 왕복 횟수를 줄이는 게 중요하다.

## 실제 사례

- **코드 에이전트**: 사용자가 "이 CSV 파일 분석해줘"라고 하면, 에이전트가 파일 읽기 → 코드 작성 → 실행 → 결과 해석의 전 과정을 자율적으로 처리.
- **리서치 에이전트**: 논문 주제를 주면, 웹 검색 → 문서 요약 → 비교 분석 → 보고서 작성 흐름을 자동화.
- **고객 지원 에이전트**: 주문 조회, 환불 처리, FAQ 검색 등의 도구를 갖추고 고객 요청을 자율 처리.

---

## 정리

- AI Agent = LLM + 도구 호출 + 반복적 추론 루프
- Tool Use는 LLM이 구조화된 출력으로 외부 함수를 호출하는 메커니즘
- ReAct 패턴으로 추론(Thought) → 행동(Action) → 관찰(Observation)을 반복
- 복잡한 태스크는 멀티 에이전트로 분산 처리
- 무한 루프, 할루시네이션, 비용 관리가 실전 구현의 핵심 과제
