# HTTP/1.1, HTTP/2, HTTP/3의 차이점

웹에서 데이터를 전송하는 프로토콜인 HTTP(하이퍼텍스트 전송 프로토콜)는 여러 버전이 존재하며, 각 버전마다 성능과 기능에서 차이를 보인다. 이번 문서에서는 HTTP/1.1, HTTP/2, HTTP/3의 주요 차이점에 대해서 정리해보겠다.

## HTTP/1.1

### 특징
- **텍스트 기반**: 요청과 응답이 모두 텍스트 형식으로 이루어짐.
- **연결 관리**: 기본적으로 매 요청마다 새로운 TCP 연결을 수립하여 성능이 떨어질 수 있다. Keep-Alive 헤더를 사용하면 연결을 유지할 수 있다.
- **비동기성 부족**: 한 연결에서 여러 요청이 동시에 이루어질 경우, 하나의 요청이 완료될 때까지 다른 요청이 대기해야 하는 문제가 있다.

### 예시
```http
GET /index.html HTTP/1.1
Host: example.com
Connection: keep-alive
```

## HTTP/2

### 개선된 점
- **이진 프로토콜**: 요청과 응답이 이진 형식으로 전송되어, 해석 및 처리 속도가 빠르다.
- **Multiplexing**: 하나의 연결에서 여러 요청과 응답을 동시에 처리할 수 있어, 대기 시간을 크게 줄일 수 있다.
- **헤더 압축**: HPACK이라는 알고리즘을 사용하여 헤더 정보를 압축해 전송 속도를 향상시킴.
- **서버 푸시(Server Push)**: 서버가 클라이언트의 요청 없이도 필요한 리소스를 미리 전송할 수 있어 로딩 속도가 향상된다.

### 예시
```http
:method: GET
:path: /index.html
:scheme: https
:authority: example.com
```

## HTTP/3

### 혁신적인 변화
- **QUIC 프로토콜**: UDP 기반의 QUIC(Quick UDP Internet Connections) 프로토콜을 사용하여 연결 지연을 최소화하고 성능을 개선한다.
- **0-RTT 핸드쉐이크**: 클라이언트와 서버 간의 연결을 빠르게 설정할 수 있어 초기 연결 속도가 크게 향상된다.
- **Multiplexing 확인**: HTTP/2의 Multiplexing 기능을 더욱 강화하여, 패킷 손실이 발생할 때 특정 연결에 대한 영향을 줄인다.

### 수신 응답 예시
QUIC을 사용하는 HTTP/3의 수신 응답은 이진 형식으로 전달되어 TCP와는 다른 방식으로 최적화된다. 실제 HTTP/3 요청 예시는 아래와 같다.

```http
GET /index.html HTTP/3
```

## 종합 비교표

| 특성           | HTTP/1.1       | HTTP/2        | HTTP/3        |
|----------------|----------------|---------------|---------------|
| 프로토콜 형식 | 텍스트 기반     | 이진 기반     | QUIC(UDP) 기반 |
| 다중 요청 처리 | 불가능 / 대기  | 가능          | 가능          |
| 서버 푸시      | 불가능         | 가능          | 가능          |
| 헤더 압축      | 없음           | HPACK 사용    | QPACK 사용    |
| 핸드쉐이크     | 1RTT           | 1RTT          | 0-RTT         |

## 결론

HTTP 프로토콜의 발전은 웹 환경의 성능 개선에 중요한 역할을 하고 있다. HTTP/1.1에서 HTTP/2, HTTP/3로의 변화는 속도와 효율성을 높였고, 다양한 기능을 통해 웹 품질을 향상시켰다. 각각의 프로토콜의 특징을 이해하고 활용하여, 웹 응용 프로그램 개발 시 더 나은 성능을 끌어낼 수 있도록 하자.
